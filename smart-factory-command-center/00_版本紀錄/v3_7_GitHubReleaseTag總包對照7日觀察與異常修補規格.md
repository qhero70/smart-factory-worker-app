# v3.7.0 GitHub Release Tag、正式總包對照、7 日觀察與異常修補規格

## 一、版本定位

```text
v3.7.0 = 正式交付與部署後維運閉環版
```

此版將 v3.6 的正式交付總包，進一步與 GitHub Release、部署後觀察、健康異常修補串成完整閉環。

完整流程：

```text
正式上線總包 ZIP
→ GitHub Release Tag 對照
→ 部署後每日健康檢查
→ 部署後 7 日觀察報告
→ 健康異常自動建立錯誤回報與修補任務
→ 持續回到修補 / 重測 / 封版流程
```

---

## 二、新增 GAS 檔案

## 三、新增資料表

### 1. 00_GitHubRelease對照

欄位：

```text
Release編號、建立時間、系統版本、GitHub倉庫、Tag名稱、Release名稱、Release狀態、Release網址、目標CommitSHA、正式總包編號、正式總包檔案、正式總包連結、備註、更新時間
```

用途：

- 對照 GitHub Release Tag 與正式上線總包。
- 保存正式總包檔案與連結。
- 若未設定 GITHUB_TOKEN，仍可產生 Release 草稿對照。

---

### 2. 00_部署後7日觀察報告

欄位：

```text
觀察編號、建立時間、系統版本、觀察起日、觀察迄日、觀察天數、健康檢查次數、正常次數、異常次數、最近健康狀態、未完成修補數、重測失敗數、觀察結論、報告內容、更新時間
```

用途：

- 彙總部署後 7 日健康檢查。
- 追蹤異常次數、未完成修補數、重測失敗數。
- 產出可交給主管的觀察報告文字。

---

### 3. 00_健康異常修補紀錄

欄位：

```text
異常編號、建立時間、來源健康紀錄編號、健康狀態、錯誤摘要、已建立錯誤回報、錯誤編號、修補編號、處理狀態、備註、更新時間
```

用途：

- 將健康檢查異常轉成錯誤回報。
- 自動建立上線修補任務。
- 避免同一健康紀錄重複開單。

## 七、GitHub Release 建立規則

如果 `00_系統設定` 有：

```text
GITHUB_TOKEN
GITHUB_REPO_FULL_NAME
```

系統會嘗試正式建立 GitHub Release。

如果沒有 `GITHUB_TOKEN`：

```text
不會中斷
會建立 Release 草稿對照
可人工複製到 GitHub Release
```

---

## 八、部署後 7 日觀察判定

觀察來源：

```text
00_部署後健康檢查紀錄
00_上線修補任務
00_修補重測清單
```

若符合：

```text
異常次數 = 0
未完成修補數 = 0
重測失敗數 = 0
```

結論：

```text
觀察穩定，可維持正式上線
```

否則：

```text
仍需追蹤異常與未完成修補
```

---

## 九、健康異常自動修補規則

掃描來源：

```text
00_部署後健康檢查紀錄
```

條件：

```text
健康狀態 = 異常
且
尚未建立過健康異常修補紀錄
```

系統動作：

```text
1. 建立 00_上線測試錯誤回報
2. 建立 00_上線修補任務
3. 寫入 00_健康異常修補紀錄
```

---

## 十、正式使用流程

```text
1. 執行 初始化_v3_0_正式上線()
2. 完成 v3.6 正式交付總包 ZIP
3. 開啟 docs/release-observe-board.html
4. 建立 Release 或 Release 草稿對照
5. 部署後每日執行健康檢查
6. 滿 7 日後產生 7 日觀察報告
7. 若健康紀錄有異常，按「健康異常開修補」
8. 回到 fix-board / retest-board 處理修補與重測
```
